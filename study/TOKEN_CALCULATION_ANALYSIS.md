# Token Calculation Analysis

This document records findings from the investigation into why `context_percentage`
reported values > 100% (e.g. 955%, 1434%, 4143%) and how the bug was fixed.

## Root Cause: Double-Counting Context Tokens

### The Problem

`aggregateTokens()` summed the `total` field from **every** assistant message.

In Anthropic sessions with prompt caching each new turn's `total` already includes
all tokens from the cached context of prior turns. Summing them counts the same
context window content N times:

```
Turn 1: total = 16,399   (all tokens processed fresh)
Turn 2: total = 16,780   (includes Turn 1's context via cache)
Turn 3: total = 22,127   (includes Turns 1+2 via cache)
...
Turn N: total = 60,430   (peak; whole prior conversation is in cache)

Naïve sum = 16,399 + 16,780 + ... + 60,430  →  WAY over 200,000
```

### The Fix: Use Peak Turn Total

The correct metric for context window usage is the **single turn with the highest
`total`** (peak turn). That value already incorporates everything the model had
access to at that point (new input + output + all cached prior context).

```
context_percentage = max(turn.total) / context_limit × 100
```

New method `getPeakTurnTokens()` finds this turn and returns its full token
breakdown plus the `modelId` it ran on.

## Token Field Structure (Anthropic)

Each assistant message has a `tokens` object:

| Field           | Meaning                                                    |
|-----------------|------------------------------------------------------------|
| `input`         | Freshly processed input tokens (not from cache)            |
| `output`        | Tokens generated by the model this turn                    |
| `cache.read`    | Tokens read from the prompt cache (context from prior turns)|
| `cache.write`   | Tokens written to cache this turn                          |
| `total`         | `input + output + cache.read` — full context window size   |

`total` is the number that should be compared against the context limit.

### Cold Turn (Turn 1)

```
input   = 16,306   ← system prompt + auto-context + user text (all fresh)
cache_r = 0        ← nothing cached yet
total   = 16,399
```

Turn 1 is the best reference for estimating composition (how the input budget is
allocated between system prompt, auto-context diffs, and user text) because
everything was processed fresh — none of it came from cache.

### Warm Turns (Turn 2+)

```
Turn 2: input=172, cache_read=16,303, total=16,780
Turn N: input=194, cache_read=60,180, total=60,430  ← peak
```

The growing `cache_read` shows cumulative context accumulation. `input` in warm
turns is only the new content the user added; everything else came from cache.

## Context Limit: Model of the Peak Turn

Some sessions switch models mid-session (e.g., haiku → sonnet). Using
`lastAssistant.modelID` to look up the context limit could give the wrong limit
if the session's peak turn was on a different model.

Fix: `getContextLimit()` now uses `peakTurnTokens.modelId` (the model that ran
the peak turn) so the percentage calculation always uses the correct limit.

## Composition Estimation

`estimateComposition()` estimates how the input budget was split across:

1. **System prompt** — OpenCode's built-in instructions
2. **Auto-context** — injected diffs from `summary.diffs`
3. **User text** — the user's actual typed messages

### Approach

Use the **first assistant message** (cold turn, `cache_read = 0`) as the reference.
Its `input` tokens represent everything that was freshly processed at session start.

Approximate token counts from character counts using a 4 chars/token heuristic.

```
coldInputTokens = firstAssistant.tokens.input
userTextTokens  = min(round(userTextChars / 4), coldInputTokens)
autoCtxTokens   = min(round(autoContextChars / 4), coldInputTokens - userTextTokens)
sysPrmptTokens  = coldInputTokens - userTextTokens - autoCtxTokens
```

### Bug: `userTextChars` Was Including Assistant Responses

The original code summed the `text` field of **all** parts, including parts
belonging to assistant messages (the model's streamed responses):

```typescript
// WRONG — includes assistant text parts
const userTextChars = parts
  .filter((p) => p.data.type === "text")
  .map((p) => p.data.text?.length ?? 0)
  .reduce((a, b) => a + b, 0);
```

This inflated `userTextTokens`, which consumed the entire `coldInputTokens` budget,
leaving `systemPromptTokens = 0` and an inaccurate breakdown.

### Fix

Filter text parts by `message_id` against the set of user message IDs:

```typescript
const userMessageIds = new Set(userMessages.map((m) => m.id));
const userTextChars = parts
  .filter((p) => p.data.type === "text" && userMessageIds.has(p.message_id))
  .map((p) => p.data.text?.length ?? 0)
  .reduce((a, b) => a + b, 0);
```

`IPart` already carries `message_id` so no schema changes were needed.

## Remaining Considerations

- **Live sessions**: If the last assistant message has `total = null` (still
  streaming), `getPeakTurnTokens()` handles it via `t.total ?? t.input + t.output`.
  The in-progress turn may briefly under-report until the step finishes.
- **Composition is an estimate**: The 4 chars/token heuristic varies by content
  type (code ≈ 3.5 chars/token, prose ≈ 4–5 chars/token). Values are directionally
  correct but not precise.
- **`tokens.total` (sum) is still tracked** for billing/cost display; only the
  context window percentage calculation switched to the peak model.

## Bug: Right Panel Overcounting (~9k tokens)

### The Problem

`renderTokenPanel()` in `dashboard.ts` was mixing tokens from two different turns
as if they were non-overlapping segments of the context window:

| Row               | Source                      |
|-------------------|-----------------------------|
| System prompt     | `comp.system_prompt_tokens` — from cold turn (turn 1) estimate |
| Auto-context      | `comp.auto_context_tokens`  — from cold turn estimate |
| User messages     | `comp.user_text_tokens`     — from cold turn estimate |
| Conversation      | `peak.cache_read`           — from peak turn |
| Output            | `peak.output`               — from peak turn |
| Free space        | `limit - peak.total`        — from peak turn |

The cold-turn estimates (`sys + auto + user ≈ 15,278`) are the exact tokens that
later appear in `peak.cache_read` (74,214). They are **already included** in the
cache_read value, so listing them both as distinct rows double-counts them:

```
15,278  +  74,214  +  368  +  69,307  =  159,167  ≠  150,000  ✗
```

### The Fix: Single-Turn Breakdown

The main breakdown now uses only values from the **peak turn**, which naturally
form non-overlapping segments that sum exactly to `context_limit`:

```
peak.input  +  peak.cache_read  +  peak.output  +  free  =  context_limit  ✓
6,111       +  74,214           +  368           +  69,307 = 150,000
```

| Row               | Value                   | Meaning                          |
|-------------------|-------------------------|----------------------------------|
| New input         | `peak.input`            | Fresh tokens processed this turn |
| Conversation      | `peak.cache_read`       | All prior turns from cache       |
| Output            | `peak.output`           | Tokens generated this turn       |
| Free space        | `limit - peak.total`    | Remaining context capacity       |

The cold-turn composition estimates (`system_prompt`, `auto_context`, `user_text`)
are now shown **below** the main breakdown as a secondary informative section
("Initial input breakdown (est.)"), with percentages relative to the cold turn's
total input — making it clear they are a decomposition of "New input", not
additional segments.
